# React Web App — Spec

> **Package:** `ze-web` (new React app under `apps/ze-web/`)
> **Phase:** 43
> **Status:** Planned
> **Depends on:** Phase 40 ([40-notifications.md](40-notifications.md)), Phase 41 ([41-component-descriptors.md](41-component-descriptors.md)), Phase 42 ([42-native-ui-foundation.md](42-native-ui-foundation.md))

---

## Purpose

This phase delivers the React web client that replaces `ze-app` (Flutter) as Ze's primary
interface. The app connects to the FastAPI backend over WebSocket, renders agent responses
as React components (using component descriptors from Phase 41), and runs in the browser
with zero compilation or platform friction.

The app is single-user and self-hosted. There is no sign-up, no account, no distribution
requirement. The user opens it in any browser and configures a server URL and API key.
A Tauri desktop wrapper can be added later without any changes to the React code.

---

## Responsibilities

- Maintain a persistent WebSocket connection to the backend, with automatic reconnect.
- Render inbound `message` frames as a chat timeline — text (markdown) + zero or more
  components below it, in order.
- Render all 8 component types as React components using TypeScript types generated from
  `scripts/generate_components.py`.
- Send user messages and `ack` frames over the WebSocket.
- Load message history via `GET /api/messages` on app open / reconnect.
- Show a "Ze is thinking…" state (disable send, show typing indicator) while a response
  is in flight.
- Persist the server URL and API key in `localStorage`.
- Refresh data screens when `refresh` frames arrive from the backend.

---

## Out of Scope

- Desktop distribution (Tauri wrapper deferred to a later phase).
- Multi-user or multi-account support.
- Voice input from the browser (the backend handles audio; this is a future concern).
- Image capture (same — backend handles it).
- Component interaction callbacks beyond confirm tap — deferred.
- Offline mode — messages are loaded from the backend; no local cache beyond the session.
- Deep link URL schemes — browser URL routing replaces the `ze-app://` scheme.
- PWA / service worker — deferred.

## WebSocket Client

```ts
// src/ws/useWebSocket.ts

export function useWebSocket(config: AppConfig) {
  // Returns:
  //   frames: ReadableStream<InboundFrame> (via Zustand store)
  //   send(frame: OutboundFrame): void
  //   isConnected: boolean
}
```

### Reconnect strategy

On disconnect: exponential backoff starting at 1 s, capped at 30 s.

On reconnect: server pushes `list_unread()` frames immediately. REST history load runs in
parallel. Messages are deduplicated by ID in a `Map<string, Message>` — frames from both
sources merge safely.

### Frame protocol

```ts
// src/ws/protocol.ts

// Server → Client
type InboundFrame =
  | { type: 'message';         message: Message }
  | { type: 'edit';            id: string; text?: string; components: ComponentDescriptor[] }
  | { type: 'confirm_request'; id: string; prompt: string; actions: ConfirmAction[] }
  | { type: 'confirm_cancel';  id: string }
  | { type: 'typing' }
  | { type: 'error';           detail: string }
  | { type: 'refresh';         screen: string }
  | { type: 'pong' };

// Client → Server
type OutboundFrame =
  | { type: 'message'; text: string; thread_id?: string; context?: ScreenContext }
  | { type: 'ack';     ids: string[] }
  | { type: 'command'; name: 'cancel' | 'costs' | 'memory' | 'contacts' }
  | { type: 'ping' };

type ScreenContext = {
  screen: string;
  goal_id?: string;
};
```

## Message Rendering

### `MessageBubble`

Each message renders as a bubble containing:
1. Text (if non-null), rendered as markdown via `react-markdown`.
2. Each component in order, rendered by `componentRenderer()`.
3. Timestamp.

User messages align right, assistant messages align left.

### `componentRenderer()` factory

```tsx
// src/components/ComponentRenderer.tsx

export function componentRenderer(descriptor: ComponentDescriptor): JSX.Element {
  try {
    switch (descriptor.type) {
      case 'table':    return <TableComponent    data={descriptor} />;
      case 'metric':   return <MetricComponent   data={descriptor} />;
      case 'list':     return <ListComponent     data={descriptor} />;
      case 'timeline': return <TimelineComponent data={descriptor} />;
      case 'progress': return <ProgressComponent data={descriptor} />;
      case 'confirm':  return <ConfirmComponent  data={descriptor} />;
      case 'form':     return <FormComponent     data={descriptor} />;
      case 'card':     return <CardComponent     data={descriptor} />;
    }
  } catch {
    return <></>;
  }
}
```

Unknown component types are silently skipped. New components added to the backend before
the app is updated degrade gracefully.

### Component specs

| Component | Renders as |
|-----------|-----------|
| `TableComponent` | `<table>` inside a `div` with `overflow-auto max-h-72`; always scrollable to prevent overflow |
| `MetricComponent` | Large value text, label below, trend chip if present |
| `ListComponent` | List of items with title, subtitle, optional status badge |
| `TimelineComponent` | Vertical timeline with time label left, event card right |
| `ProgressComponent` | Vertical stepper: done=filled, active=pulsing, pending=outline |
| `ConfirmComponent` | Card with prompt text + row of buttons (primary / secondary / danger) |
| `FormComponent` | Column of inputs / selects; submit button sends values as a JSON message |
| `CardComponent` | Card with coloured left border: info=blue, warning=amber, success=green, error=red |

### `ConfirmComponent` tap behaviour

Tapping an action sends a `message` frame with `text = action.value`. The `ConfirmComponent`
disables itself after the first tap to prevent double-submission.

## Contextual Overlay

A FAB on every non-chat screen opens a mini-chat panel anchored to the current screen
context. `Cmd+K` / `Ctrl+K` triggers the same overlay from any screen.

### Purpose

The overlay lets the user interact with Ze without leaving the current screen. Ze knows
which screen is active, acts on it, and the screen updates live beneath the overlay.

```
┌─────────────────────────────────────┐
│  Goals                              │
│  ┌──────────────────────────────┐   │
│  │ ● Write copy       done      │   │
│  │ ● Build landing    active    │   │
│  │ ○ Launch           pending   │   │
│  └──────────────────────────────┘   │
├─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┤  ← overlay slides up
│  Ze: Added "Set up analytics". Done?│
│  [ Add another milestone...   ] [➤] │
│                          × dismiss  │
└─────────────────────────────────────┘
```

### Behaviour

- **Open:** FAB tap or `Cmd+K` → overlay slides up (40% viewport height, CSS transition).
- **Context:** `OutboundFrame.context` is set to the current route and any focused
  entity ID (e.g. `{ screen: "goals", goal_id: "abc-123" }`).
- **Live update:** when a `refresh` frame arrives for the current screen, TanStack Query's
  `invalidateQueries()` triggers a background refetch. The underlying screen updates while
  the overlay stays open.
- **History:** overlay conversations are full messages stored in the backend — they appear
  in the Chat tab. The overlay is an alternate input surface, not a separate context.
- **Dismiss:** click outside, press `Esc`, swipe down on mobile, or auto-dismiss after
  10 s of inactivity following a response.
- **Follow-up:** overlay stays open after Ze responds until explicitly dismissed.
- **Thinking state:** input disabled + typing indicator while Ze processes. FAB shows a
  pulsing ring while a response is in flight.

### Screen context values

| Screen | `context` sent |
|--------|---------------|
| Goals (list) | `{ screen: "goals" }` |
| Goals (goal focused) | `{ screen: "goals", goal_id: "<uuid>" }` |
| News | `{ screen: "news" }` |
| Reminders | `{ screen: "reminders" }` |
| Contacts | `{ screen: "contacts" }` |
| Costs | `{ screen: "costs" }` |

### `refresh` frame handling

```ts
// src/navigation/RefreshHandler.tsx

function handleRefreshFrame(screen: string, queryClient: QueryClient) {
  const keyMap: Record<string, string[]> = {
    goals:     ['goals'],
    reminders: ['reminders'],
    contacts:  ['contacts'],
    costs:     ['costs'],
    news:      ['news'],
  };
  const key = keyMap[screen];
  if (key) queryClient.invalidateQueries({ queryKey: key });
}
```

- `edit` frames replace the matching message in-place by ID.
- `confirm_request` frames insert a transient confirm bubble (not persisted).

## First-Launch Onboarding

Triggered once when `localStorage` is empty. A three-step flow before the chat screen:

```
Step 1 — Welcome
  "Ze is your personal AI assistant."
  [Get started →]

Step 2 — Connect
  Server URL field  (hint: "http://localhost:8000 or your server address")
  API key field     (hint: "Find ZE_API_KEY in your backend .env file")
  [Test connection] → shows ✓ or error inline
  [Continue →]      (disabled until connection test passes)

Step 3 — Notifications (optional)
  "Install the ntfy app to receive notifications when you're away."
  Topic URL shown (copied from server config response)
  [Open ntfy] / [Skip for now]
```

After onboarding completes, the chat screen is shown and config is persisted to
`localStorage`. Onboarding never shows again unless the user taps "Reset" in settings.

## Message History Load

On app open (or after settings change):

```
1. Connect WebSocket → server pushes list_unread() immediately
2. In parallel: GET /api/messages?since=<7 days ago>&limit=200
3. Deduplicate by ID in a Map<string, Message>
4. Render messages in ascending order, scroll to bottom
5. Send { type: "ack", ids: [...] } for all unread assistant message IDs
```

A "Load earlier messages" button at the top fetches the preceding 7-day window on demand.

## TypeScript Component Types (Code Generation)

`scripts/generate_components.py` now emits TypeScript interfaces instead of Dart `@freezed`
classes. The `Makefile` target becomes:

```makefile
generate-components:
	uv run scripts/generate_components.py
```

The script outputs `apps/ze-web/src/components/types.ts`:

```ts
// Generated — do not edit manually

export interface ListItem {
  text: string;
  subtext?: string;
  status?: string;
}

export interface TableComponent {
  type: 'table';
  headers: string[];
  rows: string[][];
  title?: string;
  caption?: string;
}

// ... (all 8 component types + sub-item types)

export type ComponentDescriptor =
  | TableComponent
  | MetricComponent
  | ListComponent
  | TimelineComponent
  | ProgressComponent
  | ConfirmComponent
  | FormComponent
  | CardComponent;
```

The discriminated union on `type` enables exhaustive `switch` in `componentRenderer()`.
Committing the generated file is required — the Vite build does not run the Python script.

## Open Questions

- [ ] **`localStorage` for API key.** `localStorage` is readable by any JS on the page.
  For a self-hosted single-origin app this is acceptable, but worth noting. If the user
  serves Ze web over HTTPS from a private domain, the risk is minimal.
- [ ] **`Cmd+K` keybind.** Register via `useEffect` + `window.addEventListener('keydown')`.
  No library required for a single keybind.
- [ ] **Mobile layout.** The sidebar collapses to a bottom nav bar at `< 768px`. shadcn
  provides the sheet primitives needed for the overlay on mobile.
