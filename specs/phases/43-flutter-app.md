## Purpose

This phase delivers the Flutter client that replaces Telegram as Ze's primary interface.
The app connects to the FastAPI backend over WebSocket, renders agent responses as native
components (Phase 41), and receives background push notifications via ntfy (Phase 40).

The app is single-user and self-hosted. There is no sign-up, no account, no App Store
distribution requirement. It is installed directly on the user's device (TestFlight /
Android sideload / `flutter run`). Configuration is minimal: server URL and API key.

---

## Responsibilities

- Maintain a persistent WebSocket connection to the backend, with automatic reconnect.
- Render inbound `message` frames as a chat timeline — text + zero or more components
  below it, in order.
- Render all 8 component types as native Flutter widgets using the generated Dart classes
  from Phase 41's `make generate-components`.
- Send user messages and `ack` frames over the WebSocket.
- Load message history via `GET /api/messages` on app open / reconnect.
- Register the `ze://` URL scheme and navigate to the correct screen when a notification
  is tapped.
- Show a "Ze is thinking…" state (disable send, show typing indicator) while a response
  is in flight.
- Persist the server URL and API key in secure local storage.

---

## Out of Scope

- App Store / Play Store distribution.
- Multi-user or multi-account support.
- Voice input (the backend already handles audio transcription; this is a future UI concern).
- Image capture from the app (same — backend handles it).
- Component interaction callbacks beyond `confirm_request` tap (forms, etc.) — deferred.
- Offline mode / full local message cache — messages are loaded from the backend; local
  cache is session-scoped only.
- Custom themes or dark mode toggle — system theme is followed automatically.

## WebSocket Client

```dart
// src/ws/ws_client.dart

class ZeWebSocketClient {
  ZeWebSocketClient({required AppConfig config});

  /// Emits decoded InboundFrame objects.
  Stream<InboundFrame> get frames;

  /// Connect (or reconnect). Safe to call multiple times.
  Future<void> connect();

  /// Send an outbound frame.
  void send(OutboundFrame frame);

  /// True when the socket is open and authenticated.
  bool get isConnected;

  void dispose();
}
```

### Reconnect strategy

- On disconnect: exponential backoff starting at 1 s, capped at 30 s.
- On reconnect: WS connects first; server pushes `list_unread()` frames immediately.
  REST history load runs in parallel. `MessageRepository` deduplicates by message ID
  (stored in a `LinkedHashMap<String, Message>`) — frames from both sources are merged
  safely.
- `last_seen` is the `created_at` timestamp of the last received message, held in memory
  for the session. History is always reloaded fresh on app launch.

### Frame protocol

```dart
// src/ws/ws_protocol.dart

sealed class InboundFrame {}

class MessageFrame extends InboundFrame {
  final Message message;
}
class EditFrame extends InboundFrame {
  final String id;
  final String? text;
  final List<Map<String, dynamic>> components;
}
class ConfirmRequestFrame extends InboundFrame {
  final String id;
  final String prompt;
  final List<ConfirmAction> actions;
}
class ConfirmCancelFrame extends InboundFrame {
  final String id;
}
class TypingFrame extends InboundFrame {}
class ErrorFrame extends InboundFrame {
  final String detail;
}
class RefreshFrame extends InboundFrame {
  final String screen;   // "goals" | "news" | "reminders" | "chat" | ...
}
class PongFrame extends InboundFrame {}


sealed class OutboundFrame {}

class SendMessageFrame extends OutboundFrame {
  final String text;
  final String? threadId;
  final Map<String, String>? context;  // screen context from overlay or chat
}
class AckFrame extends OutboundFrame {
  final List<String> ids;
}
class CommandFrame extends OutboundFrame {
  final String name; // "cancel" | "costs" | "memory" | "contacts"
}
class PingFrame extends OutboundFrame {}
```

## Message Rendering

### `MessageBubble`

Each message renders as a bubble containing:
1. Text (if non-null), rendered as markdown via a lightweight inline parser.
2. Each component in order, rendered by `componentWidget()`.
3. Timestamp + read indicator.

User messages align right, assistant messages align left — standard chat UI convention.

### `componentWidget()` factory

```dart
// src/components/component_widget.dart

Widget componentWidget(Map<String, dynamic> json) {
  try {
    final descriptor = componentFromJson(json);
    return switch (descriptor) {
      TableComponent c    => TableWidget(component: c),
      MetricComponent c   => MetricWidget(component: c),
      ListComponent c     => ListWidget(component: c),
      TimelineComponent c => TimelineWidget(component: c),
      ProgressComponent c => ProgressWidget(component: c),
      ConfirmComponent c  => ConfirmWidget(component: c),
      FormComponent c     => FormWidget(component: c),
      CardComponent c     => CardWidget(component: c),
    };
  } catch (_) {
    // Unknown or malformed component — render nothing rather than crash.
    return const SizedBox.shrink();
  }
}
```

Unknown component types are silently skipped. New components added to the backend before
the app is updated degrade gracefully.

### Component widget specs

| Widget | Renders as |
|--------|-----------|
| `TableWidget` | `DataTable` inside `ConstrainedBox(maxHeight: 280)` with both horizontal and vertical `SingleChildScrollView`; always constrained to prevent overflow |
| `MetricWidget` | Large value text, label below, trend chip if present |
| `ListWidget` | `ListView` of tiles with title, subtitle, optional status chip |
| `TimelineWidget` | Vertical timeline with time label left, event card right |
| `ProgressWidget` | Vertical stepper: done=filled circle, active=pulsing, pending=outline |
| `ConfirmWidget` | Card with prompt text + row of `ElevatedButton`/`OutlinedButton`/`FilledButton(danger)` |
| `FormWidget` | Column of `TextFormField` / `DropdownButtonFormField`; submit button sends values as a JSON message |
| `CardWidget` | `Card` with coloured left border: info=blue, warning=amber, success=green, error=red |

### `ConfirmWidget` tap behaviour

Tapping an action sends a `SendMessageFrame` with `text = action.value`. The backend
receives this as a regular inbound message. The `ConfirmWidget` is replaced by a
disabled state showing which action was selected (prevent double-tap).

## Contextual Overlay

A FAB on every non-chat screen opens a mini-chat overlay anchored to the current screen
context. On macOS, `Cmd+K` triggers the same overlay from any screen.

### Purpose

The overlay lets the user interact with Ze without leaving the current screen. Ze knows
which screen is active, acts on it, and the screen updates live beneath the overlay —
without navigating to chat.

```
┌─────────────────────────────────────┐
│  Goals                              │
│                                     │
│  ┌─ Relaunch website ──────────────┐│
│  │  ● Write copy          done     ││   ← Goals screen visible underneath
│  │  ● Build landing page  active   ││
│  │  ○ Launch              pending  ││
│  └─────────────────────────────────┘│
│                                     │
├─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┤  ← overlay slides up, 40% height
│  Ze: Added "Set up analytics" after ││
│      "Launch". Anything else?       ││
│                                     ││
│  [ Add another milestone...    ] [➤]││
│                         × dismiss   ││
└─────────────────────────────────────┘
```

### Behaviour

- **Open:** FAB tap or `Cmd+K` → overlay slides up to 40% screen height via
  `DraggableScrollableSheet`.
- **Context:** `SendMessageFrame.context` is set to the current screen and any focused
  entity ID (e.g. `{"screen": "goals", "goal_id": "abc-123"}`).
- **Live update:** when a `RefreshFrame` arrives for the current screen, the Riverpod
  provider for that screen is invalidated — the underlying screen re-fetches and rebuilds
  while the overlay stays open. The user sees the change in real time.
- **History:** overlay conversations are full messages stored in the backend `messages`
  table. They appear in the Chat tab history. The overlay is an alternate input surface,
  not a separate context.
- **Dismiss:** swipe down, tap outside, press `Esc` (macOS), or auto-dismiss after 10 s
  of inactivity following a response.
- **Follow-up:** the overlay stays open after Ze responds until explicitly dismissed —
  multi-turn refinement without re-opening.
- **Thinking state:** input disabled + typing indicator while Ze processes. The FAB shows
  a pulsing ring while a response is in flight.

### Screen context values

| Screen | `context` sent |
|--------|---------------|
| Goals (list) | `{"screen": "goals"}` |
| Goals (goal focused) | `{"screen": "goals", "goal_id": "<uuid>"}` |
| News | `{"screen": "news"}` |
| Reminders | `{"screen": "reminders"}` |
| Contacts | `{"screen": "contacts"}` |
| Costs | `{"screen": "costs"}` |

### `RefreshFrame` handling

Each plugin screen's data is a Riverpod `AsyncNotifierProvider`. When a `RefreshFrame`
arrives from the WS stream, the global frame handler invalidates the relevant provider:

```dart
// src/ws/refresh_handler.dart

void handleRefreshFrame(RefreshFrame frame, WidgetRef ref) {
  switch (frame.screen) {
    case 'goals':     ref.invalidate(goalsProvider);
    case 'reminders': ref.invalidate(remindersProvider);
    case 'contacts':  ref.invalidate(contactsProvider);
    case 'costs':     ref.invalidate(costsProvider);
    case 'news':      ref.invalidate(newsProvider);
    default:          break;
  }
}
```

Provider invalidation triggers an automatic re-fetch. The screen rebuilds with fresh data
without user action.

## Deep Link Handling

The Flutter app registers `ze-app://` as its custom URL scheme. The scheme is defined as
a single constant shared by the app and the backend ntfy config:

```
# .env / config
ZE_APP_SCHEME=ze-app
```

```dart
// src/navigation/deep_link_handler.dart

const kAppScheme = 'ze-app';

void handleDeepLink(Uri uri) {
  if (uri.scheme != kAppScheme || uri.host != 'navigate') return;
  final screen = uri.queryParameters['screen'];

  // v1: all deep links navigate to chat. Dedicated screens are future scope.
  // The `screen` parameter is parsed and stored for future use but not acted on.
  router.go('/');
}
```

Deep links are handled in two cases:
- **Cold start** (app closed): `go_router`'s `redirect` receives the initial URI from
  `main.dart` and calls `handleDeepLink`.
- **Warm start** (app already open): `uni_links` stream fires in `main.dart` and calls
  `handleDeepLink`.

Both cases route to `/` (chat) in v1. When dedicated goal/cost/workflow screens are built,
the switch cases are added here without changing the protocol.

**Note on `ze-app://` vs `ze://`:** `ze` is short and risks collision with other apps
on iOS (where scheme conflicts result in undefined behaviour). `ze-app` is unique enough
for a personal app without App Store distribution.

## Message History Load

On app launch (or reconnect after settings change):

```
1. Connect WebSocket → server pushes list_unread() immediately
2. In parallel: GET /api/messages?since=<7 days ago>&limit=200
3. MessageRepository merges both sources by ID (LinkedHashMap — idempotent)
4. Render messages in ascending order, scroll to bottom
5. Send AckFrame for all unread assistant message IDs
```

The `since` window is 7 days. A "Load earlier messages" button at the top of the chat
fetches the preceding 7-day window on demand — the chat is not an infinite scroll archive
but older history is reachable.

## Implementation Notes

- **Generated classes must be committed.** `make generate-components` outputs Dart files
  to `packages/ze-app/lib/src/components/`. Commit the generated files — the Flutter
  build does not run the Python codegen script; it only runs `build_runner` for the
  `@freezed` boilerplate.
- **`riverpod` from day one.** `ZeWebSocketClient` and `MessageRepository` are Riverpod
  providers. Settings save invalidates the WS provider, triggering reconnect — no
  prop-drilling or global singleton needed.
- **`flutter_secure_storage` on macOS requires keychain entitlement.** Add
  `com.apple.security.keychain-access-groups` to `macos/Runner/Release.entitlements`.
- **WebSocket auth on iOS.** Use the `?token=` query param fallback (Phase 42 spec).
  `web_socket_channel` on iOS cannot set the `Authorization` header on the HTTP upgrade.
- **Graph-pause confirmations are non-functional in v1.** `ConfirmRequestFrame` arrives
  but tapping sends a plain message — the paused graph does not resume (it expects the
  `await_confirmation` resume path, not an inbound message). The graph times out silently.
  This is an accepted v1 constraint. Migration of the confirmation gate to the native app
  protocol is a post-Phase 43 task.
- **`FormWidget` submit format.** Values are sent as a plain message with a structured
  prefix: `[form] {"field_id": "value", ...}`. Agent instructions must be updated to
  recognise this prefix as a form reply.
- **`ze-app://` scheme.** Registered in both `ios/Runner/Info.plist` and
  `android/app/src/main/AndroidManifest.xml`. Must match `ZE_APP_SCHEME` in the backend
  ntfy config.
